# AWS deploy — account-server (Wave 4)

Scaffolding for the production stack behind `api.windyword.ai`.

**Status:** not yet applied. `terraform plan` was **not run from this machine** — Terraform and the AWS CLI aren't installed locally. Run `terraform init` + `terraform plan` on a machine with both tools configured before first apply.

---

## What gets provisioned

| Layer | Resource |
|---|---|
| Network | VPC (10.60.0.0/16), 2 public + 2 private subnets across 2 AZs, 1 NAT gateway |
| TLS | ACM cert for `api.windyword.ai`, DNS-validated via Route 53 |
| Ingress | ALB (HTTPS:443, HTTP→HTTPS 301), Route 53 A-alias `api.windyword.ai → ALB` |
| Compute | ECS Fargate cluster + service, 2 tasks (512 CPU / 1 GB each) in private subnets |
| Database | RDS Postgres 16.4 (db.t4g.micro, single-AZ dev / Multi-AZ + deletion protection on `prod`) |
| Cache | ElastiCache Redis 7.1 (cache.t4g.micro, single node) |
| Secrets | Secrets Manager holds a single JSON doc with `DATABASE_URL`, `REDIS_URL`, `JWT_SECRET`, `MFA_ENCRYPTION_KEY`, and all 5 per-consumer `*_WEBHOOK_SECRET` values |
| Observability | CloudWatch log group `/ecs/windy-<env>-account-server`, ECS Container Insights enabled |

All secrets are generated in-Terraform via `random_password` and materialized into Secrets Manager. ECS injects them into the container via `valueFrom` per-key secret references — they never appear in task-def JSON or logs.

## Prerequisites (one-time, before first apply)

1. **Route 53 hosted zone for `windyword.ai`** must exist in the same AWS account. The stack looks it up via `data.aws_route53_zone`. If you haven't moved DNS yet, create the zone first and update the domain registrar's nameservers.
2. **Container image published to ECR.** Build + push `account-server` with the existing Dockerfile:
   ```sh
   aws ecr create-repository --repository-name windy-account-server
   $(aws ecr get-login-password --region us-east-1) | docker login --username AWS --password-stdin <acct>.dkr.ecr.us-east-1.amazonaws.com
   cd account-server
   docker build -t windy-account-server:v2.0.0 .
   docker tag  windy-account-server:v2.0.0 <acct>.dkr.ecr.us-east-1.amazonaws.com/windy-account-server:v2.0.0
   docker push <acct>.dkr.ecr.us-east-1.amazonaws.com/windy-account-server:v2.0.0
   ```
3. **Remote state backend.** Un-comment and configure the `backend "s3"` block in `account-server.tf`. Create the bucket + DynamoDB lock table ahead of time:
   ```sh
   aws s3api create-bucket --bucket windy-terraform-state --region us-east-1
   aws s3api put-bucket-versioning --bucket windy-terraform-state --versioning-configuration Status=Enabled
   aws dynamodb create-table --table-name windy-terraform-lock \
     --attribute-definitions AttributeName=LockID,AttributeType=S \
     --key-schema AttributeName=LockID,KeyType=HASH \
     --billing-mode PAY_PER_REQUEST
   ```

## Apply flow

From `deploy/aws/`:

```sh
terraform init
terraform plan -out=plan.tfplan \
  -var 'container_image=<acct>.dkr.ecr.us-east-1.amazonaws.com/windy-account-server:v2.0.0'
# Review plan output carefully.
terraform apply plan.tfplan
```

Expected first-apply time: **15–25 minutes** (RDS instance creation is the long pole, ~10 min).

### What to check during plan review

- `aws_acm_certificate.api`: confirm the domain is `api.windyword.ai` (or whatever you set `api_subdomain` to).
- `aws_db_instance.main`: if `var.environment == "prod"`, `multi_az = true` and `deletion_protection = true` — don't apply to prod without confirming.
- `aws_secretsmanager_secret_version.runtime.secret_string`: Terraform will include the generated secret values in the plan output — run `plan` on a machine without untrusted observers.
- `desired_count = 2`: ALB health checks require both tasks healthy; plan will succeed but service may flap for ~2 min during first deploy while RDS finishes.

## Post-apply smoke test

```sh
curl -sSf https://api.windyword.ai/health | jq .
curl -sSf https://api.windyword.ai/.well-known/jwks.json | jq .
curl -sSf https://api.windyword.ai/.well-known/openid-configuration | jq .
```

All three should return 200 with well-formed JSON. If `/health` shows `"database":"error"` or `"jwks":"error"`, check CloudWatch logs at `/ecs/windy-prod-account-server`.

## Rotating secrets

The runtime secrets are generated by Terraform and rotating them needs a careful two-step:

1. Update `account-server.tf` to bump the rotation trigger (e.g. add `keepers = { ... }` to `random_password` blocks).
2. `terraform apply` — this writes new values to Secrets Manager.
3. Force a new ECS deployment so tasks pick up the new secrets:
   ```sh
   aws ecs update-service --cluster $(terraform output -raw ecs_cluster_name) \
     --service $(terraform output -raw ecs_service_name) --force-new-deployment
   ```

For **webhook secrets specifically**, rotating them also requires coordination with each consumer — see `deploy/docs/webhook-env-vars.md` for the zero-downtime rotation procedure.

## Tear down (non-prod only)

```sh
# Prod has deletion_protection on RDS — set via var and re-apply to remove guard first.
terraform destroy -var 'container_image=<same image tag as apply>'
```

NAT gateway + RDS storage incur charges even when idle. Destroy staging environments between use.

## What's intentionally NOT in this scaffold

- **CI/CD** — task-definition updates are manual today. Add a GitHub Actions workflow that `docker build`s, pushes to ECR, and `aws ecs update-service --force-new-deployment` on merge to `main`. Track as a Wave-5 task.
- **WAF / Shield** — recommend adding `aws_wafv2_web_acl` once you have real traffic.
- **Autoscaling** — `desired_count` is fixed at 2. Add `aws_appautoscaling_target` + CPU/memory target-tracking policy before heavy load.
- **Sentry / APM** — existing `initErrorReporting()` in `server.ts` already handles Sentry when `SENTRY_DSN` is set; plumb it through Secrets Manager when you want error tracking in prod.
- **DB migrations runner** — today the schema is built inline by `initSchema()` in SQLite mode; the scaffold provisions Postgres but the app's Postgres path is still partial. Plan a proper migration runner (e.g. `node-pg-migrate`) before cutting over from SQLite to RDS.
