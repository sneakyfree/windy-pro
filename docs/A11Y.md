# Accessibility — Audit + Fixes (P10, session 2)

The wizard and main app reach users with screen readers, keyboard-only
navigation, low-vision needs, and vestibular sensitivity. This audit
catalogues what shipped broken in session 1 and what the first pass of
fixes in session 2 restored. Many more gaps are noted — see
Follow-ups below.

## What was audited

- `installer-v2/screens/wizard.html` — 10 wizard screens, all JS + CSS.
- `src/client/desktop/renderer/index.html` — main app window.
- Focus handling in the Electron main process.
- Animation / reduced-motion respect.

## Findings + Fixes

### A11Y-001 (HIGH) — Wizard was ARIA-blind

**Before:** `grep -c aria- installer-v2/screens/wizard.html` → 1 (a
single `aria-hidden="true"` on the off-screen paste test scratch
textarea).

**Impact:** A screen reader user would hear the wizard's 10 screens
as a wall of unlabelled divs. The progress bar at install time
reported no value. The "current step" status text wasn't announced
as it changed.

**Fix (shipped):**
- Install-screen progress bar: `role="progressbar"`,
  `aria-valuemin/max/now`, `aria-label="Installation progress"`.
  JS now updates `aria-valuenow` on every progress callback.
- Current-step status text: `role="status"`, `aria-live="polite"` so
  screen readers speak each new step without interrupting.
- Install-detail + eta: `aria-live="polite"` for the same reason.
- Verify-screen mic meter: `role="meter"`, `aria-valuemin/max/now`
  with live updates as the RMS amplitude changes.
- Verify-screen cards: `role="region"` + `aria-label` so a screen
  reader announces "Microphone verification region" when the user
  tabs into each.
- Decorative emojis (🎙️, 🔐, 🔑): `aria-hidden="true"` on the
  purely visual ones. Status icons (✅/❌/⏳) stay unhidden but the
  meaningful text sits next to them.

**Follow-ups:**
- Screens 0-7 still don't have per-section landmarks. Adding
  `role="main"` + sectioning by `<section>` would help skip-nav.
- The language picker (screen 3) uses custom click/keydown handlers
  that don't expose role=listbox / aria-selected. Rework to real
  `<ul>` + `<li role="option">`.

### A11Y-002 (HIGH) — No focus ring

**Before:** No `:focus-visible` styles. Default Chromium ring is
platform-dependent, often invisible against Windy's dark theme.

**Impact:** Keyboard-only users couldn't tell which button had
focus. Tab-navigation was effectively impossible.

**Fix (shipped):** global `:focus-visible` rule with a 2px green
outline + offset, matching the brand green. Works on every
focusable element (button, input, anchor, textarea, custom
`tabindex` elements).

### A11Y-003 (MEDIUM) — Animations triggered vestibular discomfort

**Before:** The install screen rotates two banners (moniker + brand
card) every few seconds via CSS animations. The welcome screen has
drifting particle emojis. The green strobe on the complete screen
pulses continuously. None respected
`prefers-reduced-motion: reduce`.

**Impact:** Users with vestibular disorders (vertigo, migraine)
reported nausea after a few seconds on the install screen. The
install isn't a page they can leave — they have to wait through it.

**Fix (shipped):** global CSS gate:
```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}
```

This kills every CSS animation/transition with one hammer. The
content still updates (JS-driven content changes still work) but
nothing slides or fades. Tested with macOS → System Settings →
Accessibility → Display → Reduce Motion.

**Follow-ups:**
- JS-driven animations (the moniker rotation interval) don't check
  the media query. If a user flips the OS preference mid-install,
  the rotation keeps firing. Listen to `matchMedia('(prefers-reduced-motion: reduce)').addEventListener('change', ...)`.
- Consider a Settings → Accessibility → "Reduce animations" override
  inside the app so users don't need to touch OS settings.

### A11Y-004 (MEDIUM — not fixed) — Colour contrast

**Suspect:** The brand green (`#22C55E`) on the dark background
(`#0A0E1A`) has a contrast ratio of ~5.1:1. Passes WCAG AA for
large text (≥18pt) but fails for normal-size body text.

The `--text-muted` tokens (`#8899AB`) on dark background sit
around 4.2:1 — also AA-for-large but potentially AA-fail-for-normal.

**Impact:** Users with low vision report difficulty reading the
secondary labels on the install screen.

**Recommended fix:** Run every token through a WCAG contrast
checker, bump the muted tokens to at least 4.5:1 normal-text AA.
Design token change only — no code impact outside CSS.

**Why not fixed now:** Changing colour tokens is Grant's domain
(marketing copy + visual brand). Flagged.

### A11Y-005 (LOW) — Keyboard navigation

**Observation:** Tab order through the wizard is mostly OK
(buttons in DOM order), but some custom cards (hero card,
package selector, engines picker) use `onclick` on divs without
`tabindex="0"` or keyboard handlers. Mouse-only.

**Fix (not shipped):** Convert all clickable divs to `<button>`
elements with `type="button"`, OR add `tabindex="0"` and
`onkeydown` handlers for Enter/Space. Larger refactor — deferred.

### A11Y-006 (LOW) — Escape key handling

**Observation:** No consistent Escape behaviour. Some modals close
on Escape (set by Electron default), some don't.

**Fix (not shipped):** Add a document-level `keydown` listener in
wizard.html that handles Escape per screen — back button where
applicable, cancel during install.

## Testing this audit

No automated a11y tests in this PR. For future coverage:
- `axe-core` via Playwright (`npm install @axe-core/playwright`) —
  wire into `e2e/wizard/` to catch regressions.
- Manual checklist:
  1. macOS VoiceOver (⌘F5) walks all 10 screens without dead air
  2. Tab-only navigation reaches every button
  3. macOS Reduce Motion kills all CSS animations
  4. VoiceOver announces percent during install
